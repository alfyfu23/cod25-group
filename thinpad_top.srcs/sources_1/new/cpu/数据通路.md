# RV32I 流水线 CPU 数据通路设计说明

本文档详细描述了当前 RV32I 处理器核心 (`rv32i_core.sv`) 的数据通路设计。该处理器采用经典的 **五级流水线 (5-Stage Pipeline)** 结构，包含取指 (IF)、译码 (ID)、执行 (EX)、访存 (MEM) 和写回 (WB) 五个阶段。

## 1. 流水线概览

处理器通过流水线寄存器将各个阶段分隔开，每个时钟周期指令向前推进一级。

*   **IF/ID Register**: 隔离取指与译码级。
*   **ID/EX Register**: 隔离译码与执行级。
*   **EX/MEM Register**: 隔离执行与访存级。
*   **MEM/WB Register**: 隔离访存与写回级。

## 2. 详细阶段设计

### 2.1 取指阶段 (IF - Instruction Fetch)
*   **功能**: 从内存中读取指令，并计算下一条指令的 PC。
*   **主要组件**:
    *   **PC 寄存器 (`pc_reg`)**: 存储当前指令地址。
    *   **Wishbone 总线接口**: 负责发起取指请求 (`BUS_IFETCH` 状态)。
*   **下一条 PC 计算**:
    *   正常情况: `PC = PC + 4`
    *   分支/跳转 (Branch/Jump): 来自 EX 阶段的 `branch_target_ex`。
    *   异常/中断 (Trap): 来自 CSR 模块的 `trap_target_pc`。
*   **传递给下一级的数据 (`if_id_reg`)**:
    *   `pc`: 当前指令地址。
    *   `instr`: 取到的 32 位指令字。

### 2.2 译码阶段 (ID - Instruction Decode)
*   **功能**: 解析指令，读取寄存器堆，生成立即数，并进行冲突检测。
*   **主要组件**:
    *   **控制器**: 解析 `opcode`, `funct3`, `funct7`，生成控制信号（如 `alu_op`, `mem_read`, `reg_write` 等）。
    *   **寄存器堆 (`rv32i_regfile`)**: 读取源寄存器 `rs1` 和 `rs2` 的值。
    *   **立即数生成器**: 根据指令类型生成 `imm_i`, `imm_s`, `imm_b`, `imm_u`, `imm_j`。
    *   **冲突检测单元 (Hazard Detection)**: 检测 **Load-Use** 冲突。
*   **传递给下一级的数据 (`id_ex_reg`)**:
    *   控制信号: `reg_write`, `mem_read`, `mem_write`, `alu_op`, `branch` 等。
    *   数据: `rs1_value`, `rs2_value` (从寄存器堆读出的原始值), `imm` (立即数)。
    *   索引: `rd`, `rs1`, `rs2` (用于后续阶段的前递检测)。
    *   PC: `pc` (用于计算跳转目标)。

### 2.3 执行阶段 (EX - Execute)
*   **功能**: 执行算术逻辑运算，计算分支目标，处理前递（Forwarding）。
*   **主要组件**:
    *   **前递单元 (Forwarding Unit)**: 解决 **RAW (Read-After-Write)** 数据相关。
        *   如果 EX 阶段需要的源操作数 (`rs1`/`rs2`) 正好是 MEM 阶段或 WB 阶段要写入的目标寄存器 (`rd`)，则直接从后续阶段“窃取”数据，而不是使用 ID 阶段读到的旧值。
        *   优先级: MEM 阶段前递 > WB 阶段前递。
    *   **ALU (算术逻辑单元)**: 执行加减、逻辑、移位等操作。
    *   **分支判断逻辑**: 比较操作数，决定是否跳转 (`branch_taken_ex`)。
    *   **CSR 写逻辑**: 准备 CSR 写数据。
*   **传递给下一级的数据 (`ex_mem_reg`)**:
    *   `alu_result`: ALU 计算结果（也是内存访问地址）。
    *   `store_data`: 需要写入内存的数据（经过前递修正后的 `rs2` 值）。
    *   `rd`: 目标寄存器索引。
    *   控制信号: `mem_read`, `mem_write`, `reg_write` 等。

### 2.4 访存阶段 (MEM - Memory Access)
*   **功能**: 读写数据内存。
*   **主要组件**:
    *   **Wishbone 总线接口**: 发起数据读写请求 (`BUS_MEM` 状态)。
    *   **对齐逻辑**: 处理 `LB`, `LH`, `SB`, `SH` 等非字对齐访问。
*   **传递给下一级的数据 (`mem_wb_reg`)**:
    *   `mem_data`: 从内存读取的数据。
    *   `alu_result`: ALU 结果（用于非访存指令的写回）。
    *   `rd`: 目标寄存器索引。
    *   `reg_write`: 写使能信号。

### 2.5 写回阶段 (WB - Write Back)
*   **功能**: 将结果写回寄存器堆。
*   **主要组件**:
    *   **写回 Mux**: 根据 `wb_src` 选择写回数据来源：
        *   `WB_SRC_ALU`: 来自 ALU 的计算结果。
        *   `WB_SRC_MEM`: 来自内存的读取数据。
        *   `WB_SRC_PC4`: `PC + 4` (用于 `JAL`/`JALR`)。
        *   `WB_SRC_CSR`: CSR 读取结果。
    *   **寄存器堆写端口**: 将数据写入 `rd`。

## 3. 冲突与冒险处理 (Hazard Handling)

### 3.1 数据冒险 (Data Hazards)
*   **RAW (Read-After-Write)**:
    *   **处理方式**: **前递 (Forwarding)**。
    *   **逻辑**: 在 EX 阶段，通过比较 `id_ex_reg.rs1/rs2` 与 `ex_mem_reg.rd` 和 `mem_wb_reg.rd`，判断操作数是否在流水线前方。如果是，则通过 Mux 选择前递数据 (`ex_mem_forward_value` 或 `mem_wb_forward_value`) 代替寄存器堆读出的值。
*   **Load-Use Hazard**:
    *   **场景**: 上一条指令是 Load，当前指令需要使用 Load 的结果。
    *   **处理方式**: **流水线停顿 (Stall)**。
    *   **逻辑**: 在 ID 阶段检测到 `id_ex_reg.mem_read` (上一条指令是 Load) 且目标寄存器 `rd` 与当前指令源寄存器 `rs1` 或 `rs2` 冲突。
    *   **动作**:
        1.  保持 PC 不变 (`stall_if`).
        2.  保持 IF/ID 寄存器不变 (`stall_id`).
        3.  向 ID/EX 寄存器插入气泡 (Bubble/NOP)。
    *   **结果**: 消耗一个周期，等待 Load 指令进入 WB 阶段，数据即可通过前递获得。

### 3.2 控制冒险 (Control Hazards)
*   **场景**: 分支指令 (`BEQ`, `BNE` 等) 或跳转指令 (`JAL`, `JALR`) 改变了程序流。
*   **处理方式**: **流水线冲刷 (Flush)**。
*   **逻辑**: 分支结果在 EX 阶段计算得出 (`branch_taken_ex`)。
*   **动作**:
    *   如果分支成立 (`branch_taken_ex == 1`)：
        1.  将 PC 更新为跳转目标 (`branch_target_ex`)。
        2.  **冲刷 (Flush)** IF/ID 和 ID/EX 寄存器（将 `valid` 位置 0）。这意味着紧跟在分支后面的两条已经被取指/译码的指令被丢弃。
*   **性能影响**: 每次分支跳转会损失 2 个时钟周期（因为是在 EX 阶段才决定跳转）。

## 4. 总线仲裁
由于取指 (IF) 和访存 (MEM) 都需要访问 Wishbone 总线，且当前设计只有一个 Wishbone Master 接口，因此存在结构冲突。
*   **仲裁策略**: **数据优先**。
*   **逻辑**: 如果 MEM 阶段需要访问内存 (`mem_request_active`)，则总线状态机进入 `BUS_MEM` 状态，暂停取指。只有当数据访问完成后，才允许进行取指 (`BUS_IFETCH`)。这实际上会造成流水线在访存指令时的停顿。
